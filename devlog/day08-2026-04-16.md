### DB 모드 결과 (어제 측정)

| 회차 | P95 | avg | 에러율 |
|------|-----|-----|--------|
| RDB-01 | 2.99s | 1.77s | 0.85% |
| RDB-02 | 2.96s | 1.71s | 0.00% |
| RDB-03 | 2.68s | 1.51s | 0.00% |

- 조건: 50 VUs, 30s, 로그인 반복
- 결론: p(95) 2.68~2.99s — threshold `p(95)<500ms` 전부 실패

### Redis 모드 결과 (오늘 측정)

| 회차 | P95 | avg | 에러율 |
|------|-----|-----|--------|
| Redis-01 | 2.85s | 1.66s | 0.00% |
| Redis-02 | 2.61s | 1.47s | 0.00% |
| Redis-03 | 2.52s | 1.43s | 0.00% |

- 조건: 50 VUs, 30s, 로그인 반복
- 결론: DB 모드와 거의 차이 없음 → 병목이 Redis/DB가 아님

### DB vs Redis 비교 결론 (로그인 API)

| 모드 | P95 최저 | avg 최저 | 에러율 |
|------|---------|---------|--------|
| DB | 2.68s | 1.51s | 0.00% (RDB-03 기준) |
| Redis | 2.52s | 1.43s | 0.00% (Redis-03 기준) |

**Redis가 빠르지 않은 이유 — bcrypt 병목**

로그인 API 흐름:
```
1. email로 DB 유저 조회       → 수ms
2. bcrypt 패스워드 검증        → 200~300ms  ← 여기가 병목
3. Refresh Token 저장         → DB: 수ms / Redis: ~1ms
4. JWT 발급                   → 수ms
```

bcrypt는 의도적으로 느리게 설계된 해싱 알고리즘이다. 비밀번호를 빠르게 검증할 수 있으면 공격자도 빠르게 brute force할 수 있기 때문.
Refresh Token 저장이 전체 응답시간에서 차지하는 비중이 워낙 작아서 Redis로 바꿔도 P95 차이가 거의 없다.

**Redis가 빛을 발하는 곳 — `/api/auth/refresh`**

토큰 재발급 흐름:
```
1. Refresh Token 조회         → DB: 디스크 I/O / Redis: 메모리 조회
2. 새 Access Token 발급       → 수ms
```

bcrypt가 없고 토큰 조회만 한다. 여기서 DB vs Redis 차이가 극명하게 나온다.
→ refresh 엔드포인트로 별도 k6 테스트 진행

### refresh API — DB vs Redis 비교

**Redis 모드 결과**

| 회차 | P95 | avg | 처리량 |
|------|-----|-----|--------|
| Redis-01 | 13.51ms | 5.28ms | 400/s |
| Redis-02 | 5.47ms | 2.81ms | 410/s |
| Redis-03 | 5.64ms | 2.95ms | 411/s |

**DB 모드 결과**

| 회차 | P95 | avg | 처리량 |
|------|-----|-----|--------|
| RDB-01 | 54.1ms | 15.08ms | 348/s |
| RDB-02 | 6.75ms | 3.36ms | 404/s |
| RDB-03 | 5.92ms | 3.34ms | 404/s |

- 조건: 50 VUs, 30s, setup에서 100명 로그인 후 refreshToken으로 재발급 반복
- 모두 threshold `p(95)<200ms` 통과

**분석**

RDB-01이 54ms로 튄 이유: setup에서 100명 로그인 → DB에 토큰 100개 INSERT가 한꺼번에 터지면서 첫 번째 테스트만 영향받음. RDB-02/03은 워밍업 후 안정화.

안정화된 RDB-02/03 기준으로 DB(~6ms)와 Redis(~5ms)가 거의 차이 없음.

**왜 refresh에서도 차이가 크지 않나**

refresh_tokens 테이블의 token 컬럼에 인덱스가 걸려있어서 DB 조회가 빠름.
50VU 수준에서는 HikariCP 커넥션 10개로 충분히 처리 가능 → 커넥션 고갈 없음.

**Redis 전환의 실제 이점은 다른 곳에 있다**

- 동시 접속자 수백~수천으로 늘어날 때 HikariCP 커넥션 고갈 방지
- TTL 자동 만료 → DB는 만료 토큰이 계속 쌓임
- DB 장애 시 Redis는 독립적으로 동작

**Grafana HikariCP 모니터링 — DB 모드 refresh 테스트 중**

![HikariCP 커넥션풀 사용 현황](https://github.com/user-attachments/assets/ad721893-fe5b-482b-94a0-a2379cdf254a)

50VU 동시 요청에서 Active 커넥션 최대 4개만 사용됨. refresh가 ~3ms로 너무 빨리 끝나서 커넥션을 잡고 있는 시간이 거의 없어 10개 중 4개만으로 충분히 소화됨. Pending 대기 0.

**다음 실험: HikariCP 커넥션풀 2개로 줄이면?**

현재 커넥션풀 10개 → 2개로 줄이고 동일 테스트 시 Pending 대기 폭발 예상.
DB 모드: 커넥션 2개로 50VU 처리 불가 → P95 급등
Redis 모드: refresh가 DB 커넥션 안 써서 영향 없음
→ 커넥션풀 부족 상황에서 Redis의 진가를 확인할 수 있음

## 깨달은 점 / 막혔던 것

### k6-server를 별도 VM으로 분리한 이유

로컬(Windows)에서 k6를 돌려도 되지 않냐는 의문이 생길 수 있다.

**로컬에서 돌리면 안 되는 이유:**
- k6가 CPU/메모리를 잡아먹는 게 같은 환경에서 Spring Boot에 영향을 줌
- Windows → VM 구간 네트워크 오버헤드가 섞여서 순수 서버 성능이 아닌 값이 나옴
- DB vs Redis 비교 시 조건이 오염됨

**k6-server (별도 VM) 에서 돌리면:**
- k6 리소스 완전 분리 → 측정값 오염 없음
- 내부 IP(`192.168.100.10:8080`) 직접 요청 → Cloudflare 터널, DNS 레이턴시 없음
- 순수 Spring 코드 성능만 측정 가능

**실무에서도 목적에 따라 두 가지로 나뉜다:**

| 방식 | 목적 |
|------|------|
| 내부망 테스트 (지금 방식) | 서버/코드 자체 성능 측정. DB vs Redis 같은 순수 로직 차이 비교 |
| 외부망 테스트 (도메인 경유) | 실제 유저가 겪는 응답시간 측정. CDN/Cloudflare 레이턴시 포함 |

지금은 DB vs Redis 차이가 목적이므로 내부망이 맞다.
외부망으로 돌리면 Cloudflare 레이턴시가 양쪽 다 동일하게 붙어서 차이가 희석된다.
