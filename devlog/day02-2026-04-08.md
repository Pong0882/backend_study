### Docker MySQL 포트 충돌 문제
서버 실행 시 아래 에러 발생:
```
Access denied for user 'pong'@'localhost' (using password: YES)
```
→ DB 연결 자체가 안 된 줄 알았으나, 에러 원인은 **인증 실패**가 아니라 Docker MySQL 컨테이너가 아예 안 떠 있었음

Docker Desktop에서 컨테이너 실행 시도하자 아래 에러 발생:
```
Ports are not available: exposing port TCP 0.0.0.0:3306 -> 127.0.0.1:0: 
listen tcp 0.0.0.0:3306: bind: Only one usage of each socket address 
(protocol/network address/port) is normally permitted.
```

![Docker 포트 충돌 에러](https://github.com/user-attachments/assets/82941c0a-0f8b-437e-aa9c-79a8fa20dd00)

**원인**: 로컬에 MySQL 8.0이 Windows 서비스(`MYSQL80`)로 설치되어 있었고, 컴퓨터 재시작 시 자동으로 실행되면서 3306 포트를 선점함

**확인 과정**:
```bash
netstat -ano | findstr :3306
# → PID 7888이 3306 점유 중

tasklist | findstr 7888
# → mysqld.exe (로컬 MySQL)

sc query type= all state= all | findstr -i mysql
# → 서비스명: MYSQL80
```

**해결**: 로컬 MySQL 서비스 자동시작 비활성화
```bash
sc config MYSQL80 start= disabled
net stop MYSQL80
```
→ 이후 Docker MySQL 정상 실행

### 주식 현재가 조회 구현 (GET /api/stocks/{stockCode})

**구현 내용:**
- `KisStockPriceResponse` DTO — 응답 100개 필드 중 핵심 10개만 매핑
- `StockService` — 한투 API 호출 후 `output` 필드를 `ObjectMapper.convertValue()`로 DTO 변환
- `StockController` — `@PathVariable`로 종목코드 받아서 조회

**에러 과정:**

1차 — ObjectMapper Bean 없음
```
Parameter 2 of constructor in StockService required a bean of type 'ObjectMapper' that could not be found.
```
→ Spring Boot가 Jackson을 자동 등록할 것 같았는데 안 됨 (`spring-boot-starter-webmvc` 사용으로 자동 구성 범위가 달랐을 가능성)
→ `ObjectMapper`는 외부 라이브러리 클래스라 `@Component` 붙일 수 없음
→ `AppConfig`에 `@Configuration` + `@Bean`으로 직접 등록
→ `@Bean` 등록 원리 → notes/phase-7-spring-core/spring-layers.md

2차 — DTO에 없는 필드 역직렬화 실패
```
Unrecognized field "iscd_stat_cls_code" (class KisStockPriceResponse), not marked as ignorable
```
→ 한투 응답 필드가 100개 이상인데 DTO에 10개만 정의했음
→ `@JsonIgnoreProperties(ignoreUnknown = true)` 추가로 해결

### 한투 모의투자 API 연동 시작
appkey / appsecret 발급 완료. 한투 API 포털에서 접근 토큰 발급 문서 직접 확인함.

![한투 토큰 발급 API 문서](https://github.com/user-attachments/assets/e9292a95-d5cb-4147-ae95-9380f44179fa)

**토큰 발급 방식 이해:**
- 일반 OAuth (Google 로그인 등)과 달리 사용자 개입 없이 서버끼리 키를 교환하는 `client_credentials` 방식
- appkey + appsecret → POST 요청 → access_token 발급 (24시간 유효)
- 이후 모든 API 요청 헤더에 `Authorization: Bearer {access_token}` 담아서 호출

**판단 흐름:**
- 매 요청마다 토큰을 새로 발급하면 하루 호출 횟수 제한 초과 위험 → 토큰 캐싱 필요
- 모의투자 서버 URL: `https://openapivts.koreainvestment.com:29443` (실서버와 다름)
- yml에 `kis.base-url`로 분리해서 나중에 실서버 전환 시 설정값만 바꾸면 되도록 설계

**구현 계획:**
1. `KisConfig` — appkey/appsecret/base-url yml 바인딩
2. `KisAuthService` — 토큰 발급 + 메모리 캐싱
3. `KisClient` — RestClient 기반 HTTP 요청 담당

### 한투 토큰 발급 에러 과정

**에러**
```
403 Forbidden: {"error_description":"유효하지 않은 AppKey입니다.","error_code":"EGW00103"}
```
→ `.env`가 `pong-to-rich/`에 있고 Spring은 `pong-to-rich/backend/`에서 실행되어 환경변수를 못 읽음
→ Spring은 `.env` 파일을 자동으로 읽지 않음. Docker Compose가 읽어서 컨테이너에 주입하는 구조임

**해결**: `application-local.yml`에 값 직접 입력 (`application-local.yml`은 `.gitignore` 등록돼 있어서 키 노출 없음)

**토큰 발급 성공 응답:**
```json
{"access_token_preview": "eyJ0eXAiOiJKV1QiLCJh..."}
```
→ `eyJ0eXAiOiJKV1Qi`는 JWT 형식 (`{"typ":"JWT",...}`의 Base64 인코딩)
