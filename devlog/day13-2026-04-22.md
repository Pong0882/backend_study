## 구현 내용

### 배경 — 렉 원인 파악

Stocks 페이지에서 종목 선택할 때마다 심각한 렉이 발생했다. 다른 크롬 탭까지 멈추는 수준이었음.

처음에는 서버/네트워크 문제로 의심했는데, Chrome DevTools Network 탭으로 측정해보니:

```
prices    200    xhr    59.2 kB    281ms
```

서버 응답 자체는 빠른 편이었다. 281ms / 59KB.

그다음 Performance 탭으로 Main thread time을 확인하니:

```
[unattributed]    822.7ms
pongtrader.pro    343.2ms
```

Main thread가 약 1.2초 블로킹되고 있었음. 브라우저 렌더링 문제.

**렉 원인 두 가지:**

1. **테이블 DOM 렌더링** — 3,000건을 `prices.map()`으로 전부 DOM에 그리면 `<tr>/<td>` 노드가 약 18,000개 생성되면서 메인 스레드 블로킹
2. **차트 destroy/재생성** — 종목 변경 시마다 `candleChart.current.remove()` 후 `createChart()` + `setData(3,000건)` 반복

차트 블록을 임시 주석처리하고 테스트해보니 렉이 사라짐 → 차트가 렉에 기여하고 있음을 확인.

---

### 해결 방향 결정

페이지네이션 + 차트 엔드포인트 분리로 결정.

- **테이블**: 페이지네이션 적용 (`?page=0&size=50`). 한 번에 50건만 DOM에 그림
- **차트**: 별도 엔드포인트 (`/prices/chart`) 분리. 테이블 페이지 전환과 무관하게 전체 데이터 유지

기간 버튼(1개월/3개월/1년) 방식도 고려했으나, 사용자가 휠로 줌인/줌아웃 시 범위를 벗어난 데이터가 없어지는 문제가 있음. lazy loading(스크롤로 과거 데이터 추가 로딩)이 맞는 해결책이지만 구현이 복잡해 별도 이슈로 분리.



```
GET /api/stocks/{code}/prices/chart            → 차트용 전체 데이터
```

## 결과

변경 후 Network 탭 측정:

| 항목 | 변경 전 | 변경 후 |
|------|---------|---------|
| 테이블 응답 크기 | 59.2 kB | 7.9 kB (-87%) |
| 테이블 응답 시간 | 281ms | 26ms (-91%) |
| 차트 응답 크기 | - | 399 kB (전체 데이터 별도 요청) |
| 차트 응답 시간 | - | 104ms |
| 종목 변경 시 렉 | 있음 | 없음 |

수치 해석 주의: 테이블 응답 87%/91% 개선은 서버 처리 성능이 개선된 것이 아니라 전송 데이터를 3,000건 → 50건으로 줄인 결과임. 차트는 오히려 전체 데이터를 별도로 가져와서 399 kB로 증가했으나 이는 의도된 설계.

## 깨달은 점 / 막혔던 것

- 렉이 심해서 처음에 서버 문제로 의심했는데 Network 탭으로 측정해보니 서버는 빨랐음. 추측보다 측정이 먼저
- DOM 노드 수가 많아지면 브라우저 메인 스레드가 블로킹된다는 걸 직접 체감. 3,000건 × 6컬럼 = 18,000개 노드가 다른 크롬 탭까지 멈추게 만들 수 있음
- 페이지네이션이 항상 정답은 아님. 차트처럼 전체 데이터가 필요한 경우엔 엔드포인트를 분리하는 게 맞음
- 차트 lazy loading(휠 스크롤로 과거 데이터 추가 로딩)은 실제 증권사 패턴인데 구현 복잡도가 있어서 별도 이슈로 분리

## Issue #49, #58 — 증권사 계좌 KIS API 검증 + 활성화 토글

### 버그 1 — POST /api/broker-accounts 400 에러

계좌 등록 시 400이 떴다. 처음에 appkey/appsecret이 틀린 줄 알았는데 로그를 보니:

```
WARN [증권사계좌] KIS API 키 검증 실패 — MOCK: 403 Forbidden:
{"error_code":"EGW00133","error_description":"접근토큰 발급 잠시 후 다시 시도하세요(1분당 1회)"}
```

KIS 토큰 발급 API rate limit(1분당 1회)을 초과한 것. 기존 코드는 `RestClientException` 전체를 `BROKER_ACCOUNT_INVALID_KEY(400)`으로 처리해서 "유효하지 않은 키" 오류로 잘못 안내하고 있었음.

**해결**: `HttpClientErrorException`으로 분기해서 `EGW00133` 감지 시 `BROKER_ACCOUNT_KIS_RATE_LIMIT(429)` 별도 처리.

### 버그 2 — AES 암호화 실패 500 에러

KIS 검증은 통과했는데 DB 저장 시 500:

```
ERROR AES encryption failed
[AesEncryptor] encrypt.secret-key loaded, length=30bytes
```

`AesEncryptor`가 32바이트를 강제 체크하는데 `.env`의 `ENCRYPT_SECRET_KEY`가 30자였음. 2글자 추가해서 해결.

### 버그 3 — 계좌 등록 후 항상 "비활성" 표시

계좌 등록은 됐는데 프론트에서 항상 "비활성"으로 표시됐다.

원인: Java `record`에서 `boolean isActive` 선언 시 Jackson이 JSON을 `"isActive": true`로 직렬화하는데, 프론트에서 `a.active`로 읽고 있었음. `a.active`는 항상 `undefined` → falsy → "비활성".

**판단 흐름**: 백엔드 DTO 필드명을 `active`로 바꾸는 것보다 Java 컨벤션상 `isActive`가 맞으므로 프론트를 `a.isActive`로 수정.

### 활성화 API 추가

비활성화된 계좌를 다시 활성화할 API가 없었음.

- `BrokerAccount.activate()` 도메인 메서드 추가
- `BrokerAccountService.activate()` 추가
- `PATCH /api/broker-accounts/{id}/activate` 엔드포인트 추가
- 프론트: 비활성 계좌에 초록색 활성화 버튼, 활성 계좌에 빨간색 비활성화 버튼

## Issue #34 — KIS 주문 전송 연동 (1단계)

### 설계 논의 — 트랜잭션 구조

외부 API 호출을 단일 트랜잭션으로 묶으면 두 가지 문제:
1. KIS API 대기 중 DB 커넥션 홀딩
2. KIS 주문 접수 후 DB 저장 실패 시 데이터 불일치

1단계는 의도적으로 단순하게 구현. 직접 에러를 만나면서 2단계(트랜잭션 분리) 이상으로 개선 예정.

→ 정리 필요: [외부 API 호출과 트랜잭션 분리](../notes/phase-11-db-jpa/transaction-external-api.md)

### 구현 내용

**백엔드**
- `Order` — `kisOrderNo` 필드 + `assignKisOrderNo()` 추가
- `BrokerAccount` — `accountNumber` 필드 추가 (KIS 주문 시 `CANO` 파라미터)
- `KisOrderResponse` DTO 신규 생성
- `KisOrderService` 신규 생성
  - MOCK/REAL URL + tr_id 분기
  - 계좌 appkey/appsecret으로 토큰 직접 발급 (Redis 키: `kis:order-token:{accountId}`)
  - 공용 토큰(`KisAuthService`)과 분리 이유: KIS는 토큰 발급자와 주문 요청자 appkey가 일치해야 함
- `OrderService` — KIS 주문 전송 연결 (1단계: 동기, 단일 트랜잭션)
- `ErrorCode.KIS_ORDER_FAILED` 추가

**프론트**
- `BrokerAccounts.jsx` — 계좌번호 입력 필드 추가

→ 정리 필요: [KIS 계좌별 토큰 캐싱 설계](../notes/phase-16-redis/kis-account-token-caching.md)

### 에러 — EGW00356 타겟 서버 오류 (2026-04-23 디버깅)

**1. 상황**

Issue #34 이어서 KIS 모의투자 주문 테스트. IP 등록이 필요한지 확인하려고 KIS 고객센터에 문의했더니 "KIS는 OAuth 방식이라 IP 등록 불필요"라는 답변을 받음.

**2. 에러 원문**

```
WARN [KisOrder] 주문 전송 중 오류 — 500 Internal Server Error:
{"rt_cd":"1","msg_cd":"EGW00356","msg1":"타겟 서버가 올바르지 않습니다. API 문서를 확인하세요 (타겟 서버 : 입력한 TR ID의 제일 첫 문자)"}
```

**3. 디버깅 과정**

- TR ID 구버전 → 신버전 변경: `VTTC0802U` → `VTTC0012U` (문서에서 구TR 사용 금지 확인)
- `custtype: P` 헤더 누락 → 추가
- `EXCG_ID_DVSN_CD: "KRX"` 누락 → 추가 (공식 예제 코드 참고)
- Redis 캐시 삭제 후 재시도
- `KisOrderService`에서 별도 토큰 발급 → `KisAuthService` 공용 토큰 재사용으로 변경 (동일 appkey 중복 발급 방지)
- curl로 직접 KIS 서버에 요청 → 동일 에러 확인 (코드 문제 아님)
- JWT 토큰 디코딩 → `"iss":"unogw"` (실전 게이트웨이 issuer)
- 전체 요청 로그 출력 (URL, 헤더, body 전부) → 모든 값 정상 확인
- KIS Developers 테스트베드로 직접 요청 → `EGW02004` "실전투자 도메인은 모의투자 앱키로 호출 불가" (테스트베드가 실전 서버 고정)
- KIS 공식 GitHub 예제 코드(`kis_auth.py`) 분석 → 헤더 구성 동일 확인

**4. 원인 분석**

KIS Developers 코드 챗봇에 문의한 결과:

> "서버(`openapivts`) + TR ID(`VTTC0012U`) 조합 자체는 정상입니다. 동일 조합에서 계속 에러가 나면 실제 전송 헤더값을 확인하세요."

모든 값을 로그로 찍어 확인했으나 전부 정상. 챗봇도 원인을 특정하지 못하고 고객센터(1544-5000) 정식 문의를 안내함.

**5. 결론**

코드/헤더/파라미터 모두 정상. KIS 서버 측 버그로 추정. 고객센터 정식 문의 예정.

**6. 코드 변경 사항 (디버깅 과정에서 개선된 것들)**
