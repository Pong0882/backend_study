## 구현 내용

### 1. EGW00356 원인 파악 — 토큰 발급 서버 불일치

velog #27 작성 당시 "KIS 서버 측 버그로 추정"으로 마무리했는데, 오늘 원인을 발견했다.

**원인:** `KisAuthServiceTest`에 `baseUrl`이 실전 서버(`openapi.koreainvestment.com:9443`)로 하드코딩되어 있었다. 테스트가 mock이라 실제 호출은 안 하지만, 이 설정을 그대로 yml에서 사용하다 보니 실전 서버로 토큰 발급 → 모의 서버로 주문 전송 → 서버 불일치 → `EGW00356`.

KIS는 토큰 발급 서버와 주문 서버의 도메인+포트가 완전히 일치해야 한다.

**해결:** `KisConfig`를 `mockBaseUrl` / `realBaseUrl`로 분리. 계좌 타입(`MOCK`/`REAL`)에 따라 토큰 발급과 주문 서버를 일치시켰다.

### 2. KisConfig / KisAuthService 구조 개선

공용 appkey(시세 조회용)와 계좌별 appkey(주문용) 분리:

- `getAccessToken()` — `KisConfig` 공용 appkey, Redis key `kis:token:common`
- `getAccessToken(BrokerAccount)` — 계좌별 appkey, Redis key `kis:token:{accountId}`

계좌 타입별 URL 분기:
- MOCK → `openapivts.koreainvestment.com:29443`
- REAL → `openapi.koreainvestment.com:9443`

`KisAuthService`에 `EGW00133`(rate limit) 예외 처리 추가.

### 3. EGW00202 원인 파악 및 해결 — chunked 인코딩

EGW00356 해결 후 `EGW00202 GW라우팅 오류` 발생.

**원인:** Spring RestClient에서 `Map`을 그대로 `.body(map)`으로 넘기면 `Transfer-Encoding: chunked`로 전송된다. KIS 게이트웨이가 chunked를 처리하지 못해서 라우팅 단계에서 거부한다.

curl은 자동으로 `Content-Length`를 명시해서 이 문제가 없었다. 그래서 curl 테스트에서는 정상 동작한 것.

**해결:** `Map` → `byte[]`로 직렬화해서 body에 전달. `byte[]`는 크기가 고정되어 있으므로 Spring이 `Content-Length`를 자동으로 설정해준다.

```java
byte[] bodyBytes = objectMapper.writeValueAsBytes(bodyMap);
restClient.post().body(bodyBytes)
```

**결과:**
```
[KisOrder] 주문 실패 — 모의투자 장종료 입니다.
```

`EGW00202` 사라짐. KIS 게이트웨이 통과 확인. 장 중 테스트 예정.
