### Swagger Mixed Content 버그 픽스

**문제:**
Swagger UI(`https://pongtrader.pro/swagger`)에서 API 호출 시 `Failed to fetch` 에러 발생.

**원인:**
Cloudflare가 HTTPS로 요청을 받아서 Spring에 HTTP로 전달하는 구조 때문에, SpringDoc이 서버 URL을 `http://pongtrader.pro`로 자동 인식했다. 브라우저는 HTTPS 페이지에서 HTTP 요청(Mixed Content)을 차단한다.

```
브라우저(https) → Cloudflare → Spring(http 8080)
                                    ↑
                          Spring은 자신이 HTTP라고 인식
                          → Swagger 서버 URL을 http:// 로 설정 → 브라우저 차단
```

**해결:**
`SwaggerConfig`에 `server-url`을 환경변수로 주입받아 명시적으로 설정.

```java
@Value("${springdoc.server-url:http://localhost:8080}")
private String serverUrl;

return new OpenAPI()
        .addServersItem(new Server().url(serverUrl))
        ...
```

`application-prod.yml`에 추가:

```yaml
springdoc:
  server-url: https://pongtrader.pro
```

- local 환경 → 기본값 `http://localhost:8080`
- prod 환경 → `https://pongtrader.pro`

### 전역 예외 처리 구현

**문제:**
기존에는 예외가 터지면 Spring이 알아서 500으로 뭉개버렸다. 중복 이메일인지, 비밀번호가 틀린 건지 클라이언트가 알 수 없었다.

**구조:**

```
exception/
├── ErrorCode.java                 ← 예외 이름 + 상태코드 + 메시지 한 곳에서 관리
├── BusinessException.java         ← 모든 커스텀 예외의 부모 (RuntimeException 상속)
├── ErrorResponse.java             ← 응답 포맷 { code, message }
├── GlobalExceptionHandler.java    ← @RestControllerAdvice
└── auth/
    ├── DuplicateEmailException     → 409
    ├── InvalidCredentialsException → 401
    ├── InvalidTokenException       → 401
    ├── ExpiredTokenException       → 401
    └── UserNotFoundException       → 404
```

**동작 원리:**

```
throw new DuplicateEmailException()
    ↓
예외 안에 ErrorCode.DUPLICATE_EMAIL이 박혀있음
    ↓
GlobalExceptionHandler가 BusinessException으로 한 번에 잡음
    ↓
e.getErrorCode().getStatus() → 409
e.getErrorCode().getMessage() → "이미 사용 중인 이메일입니다."
    ↓
{ "code": "DUPLICATE_EMAIL", "message": "이미 사용 중인 이메일입니다." }
```

`BusinessException`을 중간에 두는 이유 — 핸들러에서 `@ExceptionHandler(BusinessException.class)` 하나로 모든 커스텀 예외를 잡을 수 있다. 예외가 늘어도 핸들러는 건드릴 필요 없음.

나중에 주문, 주식 등 새 도메인이 생기면 `ErrorCode`에 항목 추가 + `exception/도메인명/` 폴더에 예외 클래스만 추가하면 된다.

## 깨달은 점 / 막혔던 것

### 브랜치 머지 순서가 중요하다

오늘 겪은 충돌의 원인:

```
feat/2026-04-09 커밋 → main 미머지 상태
    ↓
feat/2026-04-10 브랜치 생성 (옛날 main 기준)
    ↓
feat/2026-04-10에서 origin/main pull → 충돌
```

올바른 순서:
```
1. feat/2026-04-09 → main 머지 + 푸시
2. feat/2026-04-10 브랜치 생성
3. 작업 후 PR
```

브랜치를 만들기 전에 main을 항상 최신 상태로 유지해야 충돌을 줄일 수 있다.

### Mixed Content는 CORS가 아니다

Swagger에서 `Failed to fetch` + `CORS` 메시지가 떴는데, 실제 원인은 Mixed Content(HTTPS 페이지에서 HTTP 요청)였다. 브라우저 에러 메시지가 CORS로 뜨는 경우가 있어서 혼동하기 쉽다.

구분 기준:
- **CORS** — 다른 Origin으로 요청 시 서버가 허용 안 해줄 때
- **Mixed Content** — HTTPS 페이지에서 HTTP 리소스 요청 시 브라우저가 차단
- **Postman/curl에서는 둘 다 안 걸림** — 브라우저에서만 발생하는 제약

### 커스텀 예외를 도메인별로 분리하는 이유

`BusinessException(ErrorCode.DUPLICATE_EMAIL)`로 바로 던져도 동작은 한다. 하지만 스택트레이스에 `BusinessException`만 보이면 어느 도메인에서 터진 건지 이름만으로 파악이 안 된다. `DuplicateEmailException`이라는 이름이 있으면 로그만 봐도 즉시 파악 가능하다.
