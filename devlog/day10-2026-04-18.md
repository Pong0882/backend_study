## 구현 내용 — Issue #17 KisAuthService Redis 캐싱

### 변경 이유

기존 KisAuthService는 JVM 메모리(인스턴스 필드)에 토큰을 캐싱했다.

**문제점:**
- 서버 재시작 시 토큰 날아감 → 재발급 API 호출 발생
- 추후 다중 인스턴스 환경에서 인스턴스마다 각자 발급 → KIS API 불필요 호출 증가
- TTL 관리를 코드로 직접 해야 함 (`isTokenValid()` 메서드 유지 비용)

**변경 후:**
- Redis에 `kis:token` 키로 저장, TTL 자동 만료
- 서버 재시작해도 토큰 유지
- 수평 확장 시 코드 변경 없이 공유 캐시로 동작

### 주요 변경사항

**AppConfig.java** — `RestClient` Bean 추가  
기존엔 `KisAuthService` 생성자에서 `RestClient.create()` 직접 호출했다. Bean으로 분리해야 테스트에서 Mock 주입이 가능하다.

```java
@Bean
public RestClient restClient() {
    return RestClient.create();
}
```

**ErrorCode.java** — `KIS_AUTH_FAILED` 추가

```java
KIS_AUTH_FAILED(HttpStatus.BAD_GATEWAY, "한국투자증권 토큰 발급에 실패했습니다."),
```

**KisAuthService.java** — 핵심 변경

```java
// 변경 전 — 메모리 캐싱
private String cachedToken;
private LocalDateTime tokenExpiredAt;

private boolean isTokenValid() {
    return cachedToken != null
        && tokenExpiredAt != null
        && LocalDateTime.now().isBefore(tokenExpiredAt);
}

// 변경 후 — Redis 캐싱
public String getAccessToken() {
    String cached = redisTemplate.opsForValue().get(REDIS_KEY);
    if (cached != null) {
        return cached;  // 캐시 히트
    }
    return issueToken();  // 캐시 미스
}
```

**TTL 계산:**

```java
LocalDateTime expiredAt = LocalDateTime
    .parse(response.accessTokenExpired().replace(" ", "T"))
    .minusHours(1);  // 만료 1시간 전 갱신
long ttlSeconds = ChronoUnit.SECONDS.between(LocalDateTime.now(), expiredAt);
redisTemplate.opsForValue().set(REDIS_KEY, response.accessToken(), ttlSeconds, TimeUnit.SECONDS);
```

만료 1시간 전을 TTL로 설정한 이유 — 토큰이 만료된 채로 요청이 나가는 것을 방지하기 위한 여유 시간.

## TDD 첫 적용

### 테스트 대상 분기

KisAuthService는 세 가지 분기가 있어서 TDD가 의미 있었다:

| 분기 | 기대 동작 |
|------|-----------|
| Redis에 토큰 있음 (캐시 히트) | KIS API 호출 안 하고 바로 반환 |
| Redis에 토큰 없음 (캐시 미스) | KIS API 호출 → 결과를 Redis에 저장 |
| KIS API null 응답 | BusinessException(KIS_AUTH_FAILED) throw |

### Red 확인

구현 코드를 의도적으로 망가뜨려서 테스트 3개가 모두 실패하는 것을 확인했다.

- 캐시 히트 로직 → `if (false)`로 무력화
- `issueToken()` → `return null`로 대체

→ 3개 모두 Red 확인 완료

### Green 확인

원래 구현으로 복구 후 테스트 3개 모두 통과.

### RestClient 체이닝 Mock 방법

RestClient는 메서드 체이닝 구조라 각 단계마다 Mock 객체가 필요했다.

```java
@Mock RestClient restClient;
@Mock RestClient.RequestBodyUriSpec requestBodyUriSpec;
@Mock RestClient.RequestBodySpec requestBodySpec;
@Mock RestClient.ResponseSpec responseSpec;

given(restClient.post()).willReturn(requestBodyUriSpec);
given(requestBodyUriSpec.uri(anyString())).willReturn(requestBodySpec);
given(requestBodySpec.header(anyString(), anyString())).willReturn(requestBodySpec);
given(requestBodySpec.body(any())).willReturn(requestBodySpec);
given(requestBodySpec.retrieve()).willReturn(responseSpec);
```

**해결:**
1. `build.gradle`에 `testRuntimeOnly 'org.junit.platform:junit-platform-launcher'` 추가
2. `outputs.upToDateWhen { false }` — Gradle이 test 태스크를 항상 재실행하도록 강제
3. IntelliJ `Build and run using` → **IntelliJ IDEA**로 변경

## 깨달은 점

- **TDD에서 Red 확인이 중요한 이유** — 테스트가 처음부터 통과하면 "테스트가 아무것도 검증 안 하고 있다"는 가능성을 배제할 수 없다. Red를 먼저 확인해야 "이 테스트는 실제로 실패할 수 있다"는 것을 증명한다
- **RestClient를 생성자 내부에서 직접 만들면 안 되는 이유** — `new RestClient.create()`는 테스트에서 Mock으로 교체할 수 없다. Bean으로 분리해야 `@InjectMocks`가 Mock을 주입해준다
- **StringRedisTemplate** — 토큰처럼 String 값만 저장할 때는 `RedisTemplate<Object,Object>` 대신 이것. `redis-cli`에서 바로 읽힌다

## 버그 수정 — Issue #22 Soft Delete 조회 필터 미적용

### 문제

`UserRepository`의 모든 조회 메서드에 `deleted_at IS NULL` 조건이 없었다. 탈퇴 처리된 유저가 `findByEmail`, `existsByEmail` 등에서 그대로 노출될 수 있는 상태였다.

### 해결 — `@SQLRestriction`

Repository 메서드마다 조건을 추가하는 대신 Entity 레벨에서 한 번에 걸었다.

```java
@SQLRestriction("deleted_at IS NULL")
public class User { ... }
```

JPA가 User를 조회하는 모든 쿼리에 자동으로 `AND deleted_at IS NULL`을 붙여준다.

**왜 Repository에 직접 안 넣나:**
- `findByEmailAndDeletedAtIsNull`, `existsByEmailAndDeletedAtIsNull` 처럼 메서드마다 조건을 달아야 함
- 나중에 메서드 추가 시 누락 가능성이 있음
- `@SQLRestriction`은 한 번 선언으로 모든 조회에 자동 적용 → 누락 없음

## 버그 수정 — Issue #23 회원가입 시 Portfolio 자동 생성 누락

### 문제

`AuthService.signUp()`에서 User를 저장한 후 Portfolio를 생성하는 코드가 없었다. 회원가입한 유저는 Portfolio가 없는 상태로 남았다.

### 해결

```java
@Transactional
public void signUp(SignUpRequest request) {
    // ... 기존 코드 ...
    userRepository.save(user);
    portfolioRepository.save(Portfolio.builder().user(user).build());
}
```

- `@Transactional` 추가 — User 저장과 Portfolio 저장이 하나의 트랜잭션으로 묶임. Portfolio 저장 실패 시 User 저장도 롤백됨
- `signUp()`이 기존에 `@Transactional`이 없었던 이유 — User 저장만 하는 단순한 작업이라 트랜잭션이 필요 없었음. Portfolio 저장이 추가되면서 원자성이 필요해져 추가

## 사전 수정 — DTO 유효성 검증 / StockService 정리

### 변경 이유

API 구현 전 코드 품질 개선 3가지를 먼저 처리했다.

1. **DTO 유효성 검증 추가** — `SignUpRequest`, `LoginRequest`에 `@NotBlank`, `@Email`, `@Size` 추가. `spring-boot-starter-validation` 의존성 추가. `GlobalExceptionHandler`에 `MethodArgumentNotValidException` 핸들러 추가 → 검증 실패 시 400 + 오류 메시지 반환
2. **StockService RestClient Bean 주입 전환** — 생성자 내부에서 `RestClient.create()` 직접 생성하던 방식을 `@RequiredArgsConstructor` + Bean 주입으로 변경 (테스트 Mock 주입 가능)
3. **`@Transactional(readOnly = true)` 적용** — `StockService.getDailyPrices()` 이미 적용되어 있었음 확인
