### Refresh Token DB 저장 구현

**구현한 것:**
- `RefreshToken` 엔티티 + `RefreshTokenRepository` (`refresh_tokens` 테이블)
- 로그인 시 Refresh Token DB 저장 (기존 있으면 갱신, 없으면 신규 저장)
- `POST /api/auth/refresh` — Refresh Token 검증 후 새 Access Token 발급
- `POST /api/auth/logout` — DB에서 Refresh Token 삭제

**Refresh Token 재발급 흐름:**
```
클라이언트 → API 요청 → 401 (Access Token 만료)
클라이언트 → POST /api/auth/refresh { refreshToken }
서버 → DB에서 토큰 조회 → 만료 여부 확인 → JWT 서명 검증 → 새 Access Token 발급
클라이언트 → 새 Access Token으로 원래 요청 재시도
```

**왜 Access Token은 DB에 저장 안 하는가:**
- Access Token은 수명이 짧아서 JWT 서명만으로 검증 충분 (DB 조회 없음, 빠름)
- Refresh Token은 수명이 길어서 탈취/로그아웃 대응을 위해 DB 저장 필요

**구현한 것:**
- `SwaggerConfig`에 Bearer 인증 스키마 추가 → Swagger UI 상단에 **Authorize** 버튼 생성

- DTO에 `@Schema(example = "...")` 추가 → Request Body 예시값 자동 입력

**트러블슈팅 — 로그아웃 403 + NPE:**

`JwtAuthenticationFilter`에서 SecurityContext에 `email` 문자열을 principal로 저장하는데, Controller에서 `@AuthenticationPrincipal UserDetails`로 꺼내려다 null이 됨.

```java
// 잘못된 방식
public ResponseEntity<Void> logout(@AuthenticationPrincipal UserDetails userDetails) {
    authService.logout(userDetails.getUsername());  // NPE
}

// 올바른 방식
public ResponseEntity<Void> logout() {
    Authentication authentication = SecurityContextHolder.getContext().getAuthentication();
    String email = authentication.getName();  // email 문자열 직접 꺼냄
    authService.logout(email);
}
```

### Docker 컨테이너 내부 DNS 실패로 빌드 불가

**1. 상황**

`git pull` 후 `docker compose up -d --build` 실행 시 빌드 실패.
Gradle이 Maven Central에서 의존성을 다운로드하지 못함.

**2. 에러 원문**

```
Could not GET 'https://repo.maven.apache.org/maven2/jakarta/activation/jakarta.activation-api/2.1.4/jakarta.activation-api-2.1.4.jar'.
   > repo.maven.apache.org

Exception in thread "main" java.net.UnknownHostException: services.gradle.org
BUILD FAILED in 19s
```

**3. 첫 판단**

Maven Central 서버 문제이거나 네트워크 일시 장애라고 판단했다. `--no-cache` 재시도해도 동일한 에러가 발생했다.

**4. 시도한 것들**

- `docker compose build --no-cache` → 동일 실패
- `docker system prune -f` 후 재빌드 → 동일 실패
- VM에서 `nslookup repo.maven.apache.org` → 성공 (VM 자체 네트워크는 정상)
- Docker 컨테이너 안에서 `curl https://repo.maven.apache.org` → `Could not resolve host` 실패

**5. 원인 분석**

VM의 DNS 주소가 `127.0.0.53` (systemd-resolved). 이건 VM 자기 자신을 가리키는 루프백 주소다.

Docker 컨테이너는 VM과 격리된 환경이라 `127.0.0.53`이 **컨테이너 자기 자신**을 가리키게 된다. 컨테이너 입장에서는 DNS 서버가 없는 것과 같아서 외부 도메인을 전혀 해석하지 못한다.

```
VM:                 127.0.0.53 → systemd-resolved → 외부 DNS  ✅
Docker 컨테이너:    127.0.0.53 → 자기 자신 → 없음              ❌
```

이전까지 빌드가 됐던 이유는 Docker 레이어 캐시 덕분이었다. Dockerfile에서 `build.gradle`이 바뀌지 않으면 `dependencies` 레이어를 캐시에서 그대로 쓰기 때문에 의존성 다운로드 단계를 건너뛰었다. PC 재시작 후 `--build` 옵션으로 캐시가 깨지면서 처음으로 의존성을 다운로드하려 했고 그때 DNS 문제가 드러났다.

**6. 해결 방법**

`/etc/docker/daemon.json`에 DNS 서버를 명시적으로 지정했다.

```bash
sudo mkdir -p /etc/docker
sudo tee /etc/docker/daemon.json << 'EOF'
{
  "dns": ["8.8.8.8", "8.8.4.4"]
}
EOF

sudo systemctl restart docker
```

**7. 왜 그 방법인가**

`daemon.json`은 Docker 데몬 전역 설정 파일이다. 여기에 DNS를 지정하면 이후 생성되는 모든 컨테이너가 해당 DNS를 사용한다. 구글 DNS(`8.8.8.8`, `8.8.4.4`)는 안정적이고 빠른 공개 DNS라 선택했다.

**8. 배운 점**

- Docker 컨테이너는 VM과 네트워크가 격리되어 있어서 VM에서 되는 것이 컨테이너에서 안 될 수 있다
- `127.0.0.x` 루프백 주소는 컨테이너 안에서 의미가 달라진다
- 빌드 캐시가 있으면 문제가 숨겨지다가 캐시가 깨지는 순간 터진다 — 캐시에 의존하면 안 되는 이유
- Multipass + Ubuntu 24.04 + Docker 조합에서는 `daemon.json` DNS 설정이 필수
