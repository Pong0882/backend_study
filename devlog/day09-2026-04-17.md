## 구현 내용 — Issue #11 Entity 업데이트

### 변경 이유

확정된 ERD(2026-04-16) 기반으로 기존 Entity가 누락된 컬럼이 많아 업데이트.

### User Entity

**변경 전:**
- email, password, role만 존재
- password가 NOT NULL → 소셜 로그인 불가 구조

**변경 후:**
- nickname(unique), profileImage, pointBalance(BigDecimal), loginType(Enum), isActive, deletedAt 추가
- password nullable → 소셜 로그인 사용자 대응
- createdAt, updatedAt → `@PrePersist` / `@PreUpdate`로 자동 관리
- softDelete(), updateNickname(), updateProfileImage() 메서드 추가
- LoginType Enum: LOCAL / GOOGLE / KAKAO / NAVER

**판단 흐름:**
- password를 null 허용으로 바꾼 이유 — 소셜 로그인 사용자는 비밀번호 없이 가입. users 테이블 통합 방식(2026-04-16 결정) 때문
- Soft Delete 채택 이유 — 탈퇴 유저 데이터를 바로 삭제하면 연관된 orders, strategies 등이 깨짐. deletedAt으로 논리 삭제 후 배치로 정리하는 방식이 실무 표준

### Stock Entity

**변경 전:**
- code에 단독 UNIQUE 제약

**변경 후:**
- market(KRX/NASDAQ/NYSE) Enum 추가
- UNIQUE → `(code, market)` 복합으로 변경

**판단 흐름:**
- 한국 코드(005930)와 미국 코드(AAPL)는 충돌 가능성 없지만, 같은 코드가 다른 시장에 존재할 수 있음
- 2NF 트레이드오프를 인지하고 의도적으로 stocks 테이블에 market 컬럼 유지 (2026-04-16 결정)

### StockPrice Entity

**변경 전:** 가격 4개 컬럼 `Long` 타입

**변경 후:** `BigDecimal(precision=12, scale=4)` 로 변경

**판단 흐름:**
- 미국 주식 가격은 소수점 있음 (ex. 182.5700)
- Long으로 저장하면 소수점 잘림 → BigDecimal 필수
- JPA가 BigDecimal → DB DECIMAL(12,4) 자동 매핑

### ErrorCode 추가

- `DUPLICATE_NICKNAME` — 닉네임 중복 가입 시
- `STOCK_NOT_FOUND` — 종목 없을 때 (기존 `IllegalArgumentException` → `BusinessException`으로 통일)

### Repository 추가 메서드

**UserRepository:**
- `findByNickname`, `existsByNickname`

**StockRepository:**
- `findByCodeAndMarket` — (code, market) 복합 UNIQUE 기준 조회
- `existsByCodeAndMarket`
- `findAllByMarket` — 시장별 종목 목록

### AuthService / StockService 수정

- AuthService: 닉네임 중복 체크 로직 추가, `loginType=LOCAL` 세팅
- StockService: `findByCode` → `findByCodeAndMarket(KRX)` 전환, 가격 파싱 `Long.parseLong` → `new BigDecimal`, `IllegalArgumentException` → `BusinessException(STOCK_NOT_FOUND)`

### SignUpRequest / StockPriceResponse DTO 수정

- SignUpRequest: `nickname` 필드 추가
- StockPriceResponse: 가격 타입 `Long` → `BigDecimal`

## 구현 내용 — Issue #12/16 신규 Entity 작성

### 구현한 Entity 목록

### OAuthAccount

소셜 로그인 사용자의 플랫폼별 고유 ID 저장. users 통합 방식 결정에 따라 별도 테이블로 분리.

**판단 흐름:**
- 한 유저가 카카오 + 구글 둘 다 연결 가능해야 함 → users 1:N oauth_accounts
- `(provider, provider_id)` 복합 UNIQUE — 같은 카카오 계정을 두 users에 연결하는 것 차단
- createdAt만 있고 updatedAt 없음 — 소셜 계정은 연결 후 변경할 일 없음

### BrokerAccount

증권사 API 인증 키 + 예수금 동기화 정보 보관.

**판단 흐름:**
- broker(증권사) / accountType(모의/실투자) 컬럼 분리 — 같은 증권사라도 계좌 유형이 다르면 다른 키 사용
- `(user_id, broker, account_type)` 복합 UNIQUE — 같은 증권사 같은 계좌 유형 중복 등록 차단
- appkey / appsecret DB 저장 — 추후 암호화 예정, 현재는 평문 (TODO)
- balance + balanceSyncedAt — KIS API로 주기적으로 예수금 동기화, `syncBalance()` 메서드로 업데이트
- isActive — 계좌 비활성화 시 해당 전략 실행 차단 용도

### Watchlist

유저별 관심 종목 목록. 알림 희망 가격 설정 가능.

**판단 흐름:**
- `(user_id, stock_id)` 복합 UNIQUE — 같은 종목 중복 등록 차단
- alertPrice nullable — null이면 알림 미설정 상태. 0원이 아니라 null로 구분해야 의미가 명확함
- createdAt만 존재 — 관심 종목 등록 시각만 필요, 수정은 alertPrice만 바뀜

### Strategy

자동매매 전략 설정 저장. 어느 계좌로, 어느 종목을, 어떤 조건에, 몇 주 매매할지 정의.

**판단 흐름:**
- user + brokerAccount + stock 모두 연결 — 전략 실행 시 어느 계좌로 어느 종목 주문할지 바로 알 수 있음
- status: ACTIVE / INACTIVE / PAUSED 3단계 — PAUSED는 일시정지 (INACTIVE는 완전 비활성, 차이 있음)
- lastCheckedAt — 스케줄러가 이 전략을 마지막으로 체크한 시각. 너무 오래 안 체크된 전략 감지에 활용

### StrategyCondition

전략의 매수/매도 조건 저장. 하나의 전략에 여러 조건 가능 (AND 조건).

**판단 흐름:**
- indicator VARCHAR + params JSON 혼합 방식 (2026-04-16 결정)
  - indicator: 어떤 지표인지 (RSI, MACD, MA 등) — 검색/필터 용도로 VARCHAR
  - params: 지표별 파라미터 — 지표마다 구조가 달라서 JSON으로 유연하게 저장
  - 예: indicator="RSI", params=`{"period": 14, "threshold": 30}`
- 새 조건 타입이 추가돼도 DB 스키마 변경 없이 params만 바꾸면 됨

### Portfolio + Holding

포트폴리오는 유저당 1개지만 별도 테이블 유지. Holding이 실제 보유 종목 내역.

**판단 흐름:**
- Portfolio를 별도 테이블로 분리한 이유 — 향후 다중 포트폴리오 확장 시 users 테이블 수정 없이 대응 가능 (2026-04-16 결정)
- Holding.averagePrice — 여러 번 매수 시 평균 매수가 계산해서 저장. `updateQuantityAndPrice()`로 체결 시마다 갱신
- Holding.isHidden — 특정 종목을 포트폴리오 화면에서 숨기고 싶을 때. 삭제가 아니라 숨기기 (데이터 보존)

### Order + OrderExecution

주문 요청(Order)과 실제 체결 내역(OrderExecution) 분리.

**판단 흐름:**
- Order.strategy nullable — 자동매매 전략으로 생성된 주문이면 strategy 연결, 수동 주문이면 null
- filledQuantity — 부분 체결 대응. `fill()` 호출 시마다 누적, filledQuantity >= quantity면 FILLED
- OrderExecution 별도 테이블 이유 — 하나의 주문이 여러 번에 나눠 체결될 수 있음 (부분 체결). 체결 내역 각각 기록해야 정산/히스토리 가능
- PriceType(시장가/지정가) — 시장가면 price null 허용, 지정가면 price 필수. `@Column`에 nullable 기본값 적용
