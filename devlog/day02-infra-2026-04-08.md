### 2. Hyper-V VM 저장 경로 F드라이브로 변경

**왜 필요한가:**
멀티패스는 Hyper-V를 통해 VM을 만드는데, Hyper-V의 기본 VM 저장 경로가 C드라이브로 잡혀 있다.
VM 디스크 파일(`.vhdx`)은 OS + 설치한 패키지 + 데이터가 전부 쌓이는 곳이라 용량이 크게 늘어난다.
C드라이브가 꽉 차면 Windows 자체가 불안정해지므로, 여유 공간이 많은 F드라이브로 경로를 옮겨서 VM 데이터가 F드라이브에 저장되도록 설정했다.

```powershell
# 폴더 생성
mkdir "F:\Multipass\Virtual Hard Disks"
mkdir "F:\Multipass\VMs"

# 경로 변경
Set-VMHost -VirtualHardDiskPath "F:\Multipass\Virtual Hard Disks"
Set-VMHost -VirtualMachinePath "F:\Multipass\VMs"
```

- **VirtualHardDiskPath** — VM 디스크 파일(`.vhdx`). 실제 OS/데이터가 쌓임. 용량 큼
- **VirtualMachinePath** — VM 설정 파일(`.vmcx`), 스냅샷 등. 용량 작음

> 경로 변경은 이후 생성되는 VM부터 적용됨. 이미 만든 VM은 이동 안 됨

### 3. Ubuntu VM 생성

**왜 Ubuntu인가:**
실제 서버 환경은 거의 Linux다. Ubuntu는 패키지 관리(`apt`)가 편하고 커뮤니티가 커서 문제 해결이 쉽다.
멀티패스 기본 이미지도 Ubuntu라 별도 설정 없이 바로 사용 가능하다.

**왜 이 스펙인가:**
- CPU 2코어 — Spring Boot + Cloudflare 터널 동시 실행에 1코어는 부족
- 메모리 2GB — JVM 기본 힙이 최소 512MB, 여유분 포함해서 2GB
- 디스크 20GB — Java, 빌드 결과물, 로그 쌓일 공간 고려

```bash
multipass launch --name pong-server --cpus 2 --memory 2G --disk 20G
```

```bash
multipass list
# pong-server   Running   ...   Ubuntu 24.04 LTS
```

![multipass launch 실행](https://github.com/user-attachments/assets/ce4ed02f-9199-46e8-83ae-6880b0750cb7)

### 4. VM 접속

**왜 `multipass shell` 인가:**
SSH 키 설정 없이 바로 VM 안으로 들어갈 수 있는 가장 빠른 방법.
나중에 VS Code Remote SSH나 IntelliJ Remote SSH 연결할 때는 별도 SSH 키 설정이 필요하지만, 지금은 서버 초기 세팅이 목적이라 `multipass shell`로 충분하다.

```bash
multipass shell pong-server
# ubuntu@pong-server:~$
```

### 5. 패키지 업데이트 및 커널 업그레이드

**왜 필요한가:**
VM을 처음 생성하면 이미지 기준 시점 이후 쌓인 보안 패치, 버그 수정이 적용 안 된 상태다.
특히 커널 업데이트는 보안 취약점 패치가 포함되므로 서버 운영 전 반드시 적용해야 한다.

```bash
sudo apt update && sudo apt upgrade -y
```

업그레이드 후 커널이 `6.8.0-106` → `6.8.0-107` 로 올라갔고, 새 커널 적용을 위해 VM 재시작:
```bash
# VM 밖에서
multipass restart pong-server
```

### 6. Docker 설치

**왜 Java 직접 설치 대신 Docker를 선택했나:**
처음엔 Java를 직접 설치하려 했으나, 어차피 CI/CD와 블루그린 배포는 Docker 이미지 기반으로 동작한다.
Docker로 가면 환경 일관성 확보, 롤백 용이, 의존성 격리가 모두 해결되므로 처음부터 Docker로 결정했다.
(설치했던 Java는 제거)

**왜 `docker.io` 대신 공식 저장소에서 설치했나:**
Ubuntu 기본 저장소의 `docker.io` 는 버전이 오래돼서 Compose V2 플러그인이 없다.
Docker 공식 저장소를 apt에 추가해야 최신 Docker CE + Compose V2를 설치할 수 있다.

**설치 과정:**
```bash
# 1. HTTPS 통신 도구 설치
sudo apt install -y ca-certificates curl

# 2. Docker 공식 GPG 서명 키 등록 — 저장소 신뢰 확인용
sudo install -m 0755 -d /etc/apt/keyrings
sudo curl -fsSL https://download.docker.com/linux/ubuntu/gpg -o /etc/apt/keyrings/docker.asc
sudo chmod a+r /etc/apt/keyrings/docker.asc

# 3. Docker 공식 저장소 apt에 등록
sudo tee /etc/apt/sources.list.d/docker.list <<EOF
deb [arch=amd64 signed-by=/etc/apt/keyrings/docker.asc] https://download.docker.com/linux/ubuntu noble stable
EOF

# 4. 저장소 목록 갱신
sudo apt update

# 5. Docker CE + Compose V2 설치
sudo apt install -y docker-ce docker-ce-cli containerd.io docker-buildx-plugin docker-compose-plugin
```

**설치된 패키지 역할:**
- `docker-ce` — Docker 엔진 (Community Edition)
- `docker-ce-cli` — `docker` 명령어 CLI
- `containerd.io` — 컨테이너 런타임 (Docker 엔진이 내부적으로 사용)
- `docker-buildx-plugin` — 멀티 플랫폼 이미지 빌드 지원
- `docker-compose-plugin` — `docker compose` V2 명령어

```bash
docker compose version
# Docker Compose version v5.1.1
```

### 7. Cloudflare 터널 설정

**전체 흐름:**
```
Cloudflare 계정 생성
→ pongtrader.pro 도메인 추가
→ 가비아 네임서버 → Cloudflare 네임서버로 변경
→ DNS 전파 확인
→ VM에 cloudflared 설치
→ 터널 생성 및 config.yml 작성
→ DNS CNAME 레코드 등록
→ systemd 서비스 등록
```

**왜 Cloudflare 터널인가:**
로컬 PC는 공인 IP가 없거나 ISP/공유기에서 포트를 차단하는 경우가 많다.
Cloudflare 터널은 VM이 Cloudflare에 먼저 아웃바운드 연결을 맺어서 외부 포트 개방 없이 트래픽을 받을 수 있다.
추가로 DDoS 방어, HTTPS 자동 적용도 Cloudflare가 처리해준다.

```yaml
tunnel: pong-tunnel
credentials-file: /home/ubuntu/.cloudflared/{tunnel-id}.json

  - hostname: pongtrader.pro
    service: http://localhost:8080
  - service: http_status:404
```

**DNS CNAME 레코드 등록:**
```bash
cloudflared tunnel route dns pong-tunnel pongtrader.pro

```

**systemd 서비스 등록:**
```bash
sudo cloudflared --config /home/ubuntu/.cloudflared/config.yml service install
sudo systemctl enable cloudflared
sudo systemctl start cloudflared
```

## 트러블슈팅

### 1. docker-compose-plugin 패키지 못 찾음
```
E: Unable to locate package docker-compose-plugin
```
**원인:** Ubuntu 기본 저장소(`docker.io`)에는 Compose V2 플러그인이 없음  
**해결:** Docker 공식 저장소를 apt에 추가 후 재설치

### 2. docker.list URI 오류
```
E: Malformed entry 1 in list file /etc/apt/sources.list.d/docker.list (URI)
```
**원인:** 터미널에서 멀티라인 명령어 붙여넣기 시 줄바꿈이 URI 중간에 삽입됨  
**해결:** `sudo tee` + heredoc(`<<EOF`)으로 파일 직접 작성

### 3. Docker 데몬 미실행
```
Cannot connect to the Docker daemon at unix:///var/run/docker.sock. Is the docker daemon running?
```
**원인:** Docker 설치 후 데몬이 자동 시작되지 않음  
**해결:**
```bash
sudo systemctl start docker   # 즉시 실행
sudo systemctl enable docker  # 부팅 시 자동 실행 등록
```

### 4. SSH 접속 안 됨 — 외부 스위치 시도 및 롤백

**상황:** 로컬에서 VM에 SSH 직접 접속 시도했으나 타임아웃
```
ssh: connect to host 172.23.19.141 port 22: Connection timed out
```

**원인:** Hyper-V 기본 스위치(Default Switch)는 NAT 방식이라 호스트 → VM 방향 직접 접근이 안 됨

**시도 1 — Hyper-V 외부 스위치 생성:**
```powershell
New-VMSwitch -Name "ExternalSwitch" -NetAdapterName "이더넷" -AllowManagementOS $true
Get-VM -Name pong-server | Get-VMNetworkAdapter | Connect-VMNetworkAdapter -SwitchName "ExternalSwitch"
```
→ VM 재시작 후 멀티패스가 VM을 아예 못 찾는 문제 발생
```
shell failed: ssh connection failed: 'Timeout connecting to pong-server.mshome.net'
```
→ 외부 스위치와 멀티패스 충돌. 롤백

![Hyper-V 관리자 ExternalSwitch 연결 확인](https://github.com/user-attachments/assets/09c46c5e-51ee-490d-ae57-5c934da25631)

**롤백:**
```powershell
Stop-VM -Name pong-server -Force
Get-VM -Name pong-server | Get-VMNetworkAdapter | Connect-VMNetworkAdapter -SwitchName "Default Switch"
Start-VM -Name pong-server
```

**시도 2 — 멀티패스 내장 SSH 키 사용:**
멀티패스가 내부적으로 사용하는 SSH 키 발견
```
C:\ProgramData\Multipass\data\ssh-keys\id_rsa
```

키 권한 문제로 접속 실패:
```
Load key "C:/ProgramData/Multipass/data/ssh-keys/id_rsa": Permission denied
```

PowerShell에서 권한 수정:
```powershell
icacls "C:\ProgramData\Multipass\data\ssh-keys\id_rsa" /inheritance:r /grant:r "82104:R"
```

`172.23.31.44` (두 번째 IP)로 접속 성공:
```bash
ssh -i "C:/ProgramData/Multipass/data/ssh-keys/id_rsa" ubuntu@172.23.31.44
```

> `multipass list`에 IP가 여러 개 나올 때 첫 번째(`172.23.19.141`)는 Hyper-V 가상 네트워크 IP, 두 번째(`172.23.31.44`)가 실제 SSH 접속 가능한 IP였음

### 6. IntelliJ Remote SSH CPU 최소 요구사항 경고
```
The host should have at least four cores. Found: 2
```
**원인:** IntelliJ Remote SSH는 VM에 JetBrains 백엔드 서버를 띄우므로 리소스를 많이 사용
**조치:** VM 메모리 2GB → 4GB 증설. CPU는 경고 무시하고 진행
```bash
multipass stop pong-server
multipass set local.pong-server.memory=4G
multipass start pong-server
```

### 7. IntelliJ Remote SSH 연결

**SSH 키 설정:**
멀티패스가 내부적으로 사용하는 SSH 키를 활용.
키 파일 권한 문제가 있어서 PowerShell에서 권한 수정 후 접속 성공.

```powershell
icacls "C:\ProgramData\Multipass\data\ssh-keys\id_rsa" /inheritance:r /grant:r "82104:R"
```

**IntelliJ 연결 설정:**
- Host: VM IP
- Port: 22
- Username: `ubuntu`
- Private key: `C:\ProgramData\Multipass\data\ssh-keys\id_rsa`

![IntelliJ Remote SSH 접속 성공](https://github.com/user-attachments/assets/78614898-0277-4364-92d6-40b0acc5eeec)

**참고:**
- VM IP는 재시작 시 변경될 수 있음 → 매번 `multipass list`로 확인 필요
- IntelliJ Remote SSH는 VM에 JetBrains 백엔드 서버를 띄워서 메모리 사용량이 큼 → VM 메모리 4GB로 증설
- `172.23.x.x` 는 로컬 내부 IP라 외부 노출 무관
