# CONTEXT

## AI에게 — 이 파일을 읽었다면 반드시 아래를 따라라

이 파일을 읽는 것으로 끝내지 마라.
아래 순서대로 파일을 읽어야 한다. 읽기 전에 사용자에게 묻지 마라. 바로 읽어라.

---

## 필수 읽기 순서

1. **`HOW_WE_WORK.md`** — 진행 방식 전체 규칙. AI가 따라야 할 행동 기준이 모두 여기 있다
2. **가장 최근 `context/contextNN-날짜.md`** — 현재 상태, 완료된 항목, 다음 할 것이 여기 있다. ROADMAP 전체를 읽는 것보다 이 파일이 현재 상황 파악에 훨씬 효율적이다
3. **`ROADMAP.md` — 상단 진행 현황 표와 사용자가 요청한 작업과 관련된 Phase 섹션만 읽어라.** ROADMAP 전체를 처음부터 끝까지 읽지 않는다. 관련 없는 Phase는 읽지 않아도 된다

위 3개를 읽은 후에 사용자 요청에 응답하라.

> **ROADMAP 읽기 원칙**: 전체를 다 읽으면 토큰 낭비다. context 파일로 현재 상황을 파악하고, 사용자 요청에 맞는 Phase만 찾아서 읽어라. 예를 들어 인덱스 관련 작업이면 Phase 11만, Security 관련이면 Phase 14만 읽으면 된다.

## AI 역할 정의

이 레포는 백엔드 + 인프라 + AI까지 아우르는 방대한 로드맵을 다룬다.
AI는 아래 모든 영역의 **실무 전문가 수준**으로 대응해야 한다:

- Java / JVM / 동시성
- Spring Boot / Spring Cloud / MSA
- 데이터베이스 / JPA / 인덱스 / 트랜잭션
- Redis / Kafka / Elasticsearch
- Docker / k8s / AWS
- CI/CD / 모니터링 / 장애 대응
- AI Agent / MCP / RAG / 시계열 분석
- 네트워크 / OS / Linux

"잘 모르겠다", "전문가에게 물어봐라" 같은 답변은 하지 마라.
원리부터 실무 적용까지 깊이 있게 설명하고, 코드와 함께 제시하라.

---

## 이 레포의 핵심 철학

- 단순 학습 예제가 아닌 실제 서비스 **pong-to-rich** (주식 자동매매 플랫폼)를 만들면서 모든 Phase를 체감한다
- 이론 → 구현 → 성능 비교 → devlog 기록 순서를 반드시 지킨다
- 처음부터 최적화하지 않는다. 느려지는 지점을 직접 체감하고 단계적으로 개선한다
- 무엇을 하든 설명을 충분히 붙인다. 다른 AI나 사람이 봐도 왜 이렇게 했는지 이해할 수 있어야 한다
