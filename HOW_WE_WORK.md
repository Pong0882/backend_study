## 프로젝트 방향

- **hello-spring**: 개념 확인용 간단한 테스트 코드. 가볍게 쓰고 버려도 됨
- **pong-to-rich**: 실제 서비스 수준으로 키워나가는 메인 프로젝트. ROADMAP.md의 모든 Phase를 이 프로젝트에 순차적으로 적용
  - 기획 전체 내용: [pong-to-rich/PROJECT.md](./pong-to-rich/PROJECT.md)
  - 아키텍처 진화: 모놀리식 → 멀티모듈 → MSA

## 학습 사이클

각 Phase는 아래 순서로 진행한다:

```
1. 이론 정리   → notes/phase-N-xxx/*.md 에 작성
2. 구현 실습   → pong-to-rich/ 에 실제 코드로 적용
3. 성능 비교   → 적용 전/후 차이를 측정하고 기록
4. 깨달은 점   → devlog/dayNN-날짜.md 에 정리
```

> **중요**: 위 단계 중 무엇을 하든 그날의 devlog에 반드시 기록한다.
> 이론만 정리해도, 코드 한 줄만 써도, 막혔어도 — 모두 devlog에 남긴다.
> devlog는 작업 시작할 때 열고, 작업하면서 실시간으로 채워나간다.
>
> **이론 정리 페이스 규칙:**
>
> - 하루에 이론 정리는 9~10개로 제한
> - 구현 중 새로 나온 개념은 notes에 바로 정리하지 않고 devlog에 `정리 필요: 개념명` 으로 메모만 해둠
> - 나중에 사용자가 요청하면 그때 하나씩 notes에 정리

## AI가 코드를 작성할 때 체크리스트

AI가 코드를 작성하면 반드시 아래를 모두 수행한다. 하나도 빠뜨리지 마라.

### 1. devlog 기록

- 해당 날짜 `devlog/dayNN-날짜.md` 에 구현 내용 추가
- 아래 항목을 빠짐없이 기록한다:
  - 무엇을 구현했는지
  - 왜 이렇게 구현했는지 (설계 이유)
  - 어떤 선택지가 있었고 왜 이것을 골랐는지 (판단 흐름)
  - 사용자의 생각 흐름 / 진행 과정 (대화 중 나온 고민, 질문, 결정)
  - 코드에서 사용된 핵심 개념 (어노테이션, 패턴, 라이브러리 등)
  - 에러가 발생했다면 에러 원문 + 원인 + 해결 과정
  - 막혔던 부분이 있다면 그것도 기록
  - 스크린샷이 있으면 반드시 마크다운 이미지로 첨부

> devlog는 단순 작업 기록이 아니라 **판단 흐름과 생각 과정까지 담는 일지**다.
> 새로운 AI나 사람이 읽었을 때 "왜 이렇게 했는지"를 완전히 이해할 수 있어야 한다.

### 2. 사용된 개념 → notes 또는 pending 처리

코드에 등장한 모든 개념에 대해 아래 기준으로 처리한다:

```
해당 개념이 이미 notes에 정리되어 있는가?
    YES → devlog에 링크만 추가
    NO  → notes/pending.md 에 추가 (발생일 기록)
              + devlog에 "정리 필요: 개념명" 메모
```

- pending에 추가할 때는 반드시 **어느 코드에서 등장했는지** 함께 기록한다
- 사용자가 요청하면 그때 해당 Phase notes 파일에 정식 정리한다

### 3. ROADMAP 체크

- 구현한 내용이 ROADMAP의 실습 항목과 연결되면 해당 항목 체크
- ROADMAP 상단 진행 현황 표 숫자 업데이트
- context의 "완료된 항목" 섹션 업데이트

### 4. velog 초안 작성

- 작업이 마무리되면 `devlog/velog/dayNN-번호-주제.md` 에 포스팅 초안 작성
- 하루에 2개 작성: **01 — 수행한 것 요약**, **02 — 트러블슈팅**
- 파일명: `dayNN-01-주제.md`, `dayNN-02-주제.md`

**제목 형식:**
- 일반: `[pongtrader] #N — 제목`
- 트러블슈팅: `[pongtrader] #N — 트러블슈팅: 제목`
- `#N` 은 전체 포스팅 통합 번호 (day 상관없이 1부터 순서대로 증가)

**작성 스타일:**
- 독자가 읽는 블로그 포스팅 스타일 (devlog와 다르게 설명 위주)
- 에러는 **상황 → 에러 원문 → 원인 분석 → 해결** 순서로 작성

### 5. 커밋 안내

- 위 모든 작업이 끝나면 커밋할 내용과 명령어를 정리해서 사용자에게 전달
- 커밋은 목적별로 분리 (코드 / 이론 정리 / devlog / pending 업데이트)

## 대화 중 나온 개념 처리 규칙

사용자가 질문해서 AI가 설명한 내용도 학습 자산이다. 아래 기준으로 처리한다.

### 처리 기준

```
실제 구현 중에 새 개념이 등장했는가?
    YES → pending.md 에 추가 (발생일, 어느 코드에서 나왔는지 기록)
    NO  → 추가하지 않는다

단순히 대화 중 궁금해서 물어본 개념인가?
    → ROADMAP에 이미 항목이 있으면 → 추가하지 않는다 (ROADMAP이 관리)
    → ROADMAP에 없으면 → ROADMAP 항목 추가 검토만 한다
```

### 규칙

- pending은 **실제 구현하다가 나온 개념**만 추가한다
- 그냥 궁금해서 물어본 것, ROADMAP 논의 중 나온 것은 추가하지 않는다
- pending에 추가할 때는 어느 코드/파일에서 등장했는지 반드시 기록한다

## devlog 작성 규칙

- 파일명: `dayNN-YYYY-MM-DD.md` (예: `day01-2026-04-07.md`)
- 내용 구성:

```markdown
# Day N — YYYY-MM-DD

## 오늘 한 것

- ...

## 이론 정리 링크

- [주제명](../notes/phase-N-xxx/파일명.md)

## 구현 내용

- ...

## 성능 비교

- ...

## 깨달은 점 / 막혔던 것

- 에러 원문 (로그 한 줄 그대로):
```

에러 메시지 원문

```
→ 어느 부분을 보고 어떻게 판단했는지 → 어떻게 해결했는지
```

> 작업 시작할 때 파일을 열고, 진행하면서 계속 추가한다. 하루에 여러 작업을 해도 같은 파일에 이어서 작성한다.
> 에러가 발생하면 에러 로그 원문을 그대로 붙여넣고, 어느 부분을 보고 어떤 판단을 했는지, 어떻게 해결했는지 반드시 기록한다.
> 스크린샷 첨부 시 이미지 파일을 git에 직접 올리지 않는다. GitHub Issues 본문에 이미지를 드래그 앤 드롭하면 자동 링크가 생성되고(`https://github.com/user-attachments/assets/...`) 그 링크를 md에 붙여넣는다. Issue를 저장하지 않아도 링크는 영구 유효하다.
> 사용자가 결과 스크린샷을 올리면 반드시 해당 날짜 devlog에 마크다운 이미지 문법으로 추가한다.
>
> ```markdown
> ![설명](https://github.com/user-attachments/assets/...)
> ```

## notes 작성 규칙

- 각 Phase 폴더 안에 주제별 `.md` 파일로 작성
- 파일명은 영어 소문자, 하이픈 구분 (예: `ioc-di.md`, `http-basics.md`)
- 이론 정리 후 해당 날짜의 devlog에서 링크로 연결

### general 폴더 (notes/general/)

- 대화 중 궁금해서 물어봤는데 ROADMAP Phase에 없는 개념은 `notes/general/` 에 정리
- Phase와 직접 연관되지 않는 참고 지식 (명령어 모음, 용어 정리 등) 보관
- pong-to-rich 구현과 무관한 내용만. 구현 중 나온 개념은 pending으로

---

### 정리 필요 목록 (notes/pending.md)

- 구현하다가 나온 개념인데 아직 정리 못 한 것들은 `notes/pending.md` 에 추가
- 발생한 날짜(dayNN)를 함께 기록
- 사용자가 요청하면 해당 개념을 해당 Phase notes 파일에 정리하고 pending.md에서 ✅ 체크

### pending 개념 학습 플로우

```
구현 중 새 개념 등장
       ↓
pending.md 에 메모
       ↓
PR 리뷰에서 해당 코드 라인에 코멘트로 간단히 설명 (예습)
       ↓
나중에 사용자가 요청하면 notes에 정식 정리 (복습)
```

- PR 머지 전 pending 개념이 코드에 등장하면 AI가 리뷰 코멘트 내용을 작성해준다
- 사용자가 GitHub PR에 직접 붙여넣기 해서 리뷰 연습

## ROADMAP 체크리스트 업데이트 규칙

이론 정리 파일을 작성하면 반드시 아래 순서로 ROADMAP.md를 업데이트한다:

1. 해당 항목 `- [ ]` → `- [x]` 로 체크
2. 항목 끝에 `→ [정리](./notes/phase-N-xxx/파일명.md)` 링크 추가
3. **ROADMAP.md 상단 진행 현황 표 숫자 업데이트**
   - 이론 항목 체크 시: 이론 완료 +1, 미완료 -1, 전체 완료 +1, 전체 미완료 -1
   - 실습 항목 체크 시: 실습 완료 +1, 미완료 -1, 전체 완료 +1, 전체 미완료 -1
4. **가장 최근 contextNN.md 의 "완료된 항목" 섹션 업데이트**
   - 완료된 항목을 Phase별로 정리해서 기록
   - AI가 ROADMAP 전체를 읽지 않아도 현재 진행 상황을 파악할 수 있게 하기 위함

**예시:**

```markdown
- [x] **되돌리기 명령어 차이 완전 정리** → [정리](./notes/phase-5-git/git-basics.md)
```

- 정리 파일 하나가 여러 항목을 커버할 수 있음 → 각 항목마다 개별 링크
- 실습 항목(`실습:`)은 실제 코드를 작성한 후에 체크
- 체크 후 커밋은 이론 정리 커밋과 함께 묶어서 진행

## 보안 — 파일에 절대 넣으면 안 되는 정보

이 레포는 GitHub에 올라간다. 아래 정보가 포함된 파일은 커밋하지 않는다.
AI도 notes / devlog / context 작성 시 아래 정보를 절대 포함하지 않는다.

### 넣으면 안 되는 것

| 종류 | 예시 | 이유 |
|------|------|------|
| API 키 / 시크릿 | `appkey`, `appsecret`, `access_token` | 탈취 시 타인이 API 무단 사용 |
| 비밀번호 | DB 비밀번호, 서버 계정 비밀번호 | 서버/DB 무단 접근 |
| 서버 내부 IP | VM IP, 내부망 IP | 네트워크 구조 노출 |
| 도메인 + 포트 조합 | `pongtrader.pro:8080` | 공격 대상 특정 가능 |
| SSH 개인키 | `id_rsa` 내용 | 서버 무단 접근 |
| `.env` 파일 내용 | 환경변수 전체 | 위 항목 전부 포함될 수 있음 |

### 넣어도 되는 것

- 도메인 이름 단독 (`pongtrader.pro`) — 이미 공개된 정보
- 포트 번호 단독 (`8080`) — 일반적인 개발 포트
- VM 스펙 (CPU, 메모리, OS) — 공격에 활용 불가
- 에러 메시지 — 단, 경로/키 값이 포함된 경우 가려서 기록

### 민감 정보가 필요한 경우

- 코드: `application-local.yml` 또는 `.env` 에 넣고 `.gitignore` 등록
- 설명이 필요한 경우: 실제 값 대신 `{YOUR_API_KEY}` 형태로 플레이스홀더 사용

## AI가 이 레포에서 작업할 때 따라야 할 규칙

1. **작업 시작 전** — 해당 날짜의 devlog 파일이 없으면 먼저 생성한다
2. **이론 정리 작성 시** — `notes/phase-N-xxx/파일명.md` 에 작성하고, ROADMAP.md 해당 항목 체크 + 링크 추가
3. **코드 작성 시** — `pong-to-rich/` 안에서 작업. hello-spring은 간단한 테스트 용도만
4. **작업할 때마다** — devlog에 실시간으로 기록 (이론 정리, 구현, 막힌 것 모두)
5. **커밋 명령어는 직접 실행하지 않는다** — 커밋할 내용과 명령어를 알려주고 사용자가 직접 실행
6. **git 명령어도 직접 실행하지 않는다** — 명령어를 알려주고 사용자가 직접 실행
7. **모든 터미널 명령어는 직접 실행하지 않는다** — 아래 모든 경우에 해당한다. 명령어를 알려주고 사용자가 직접 실행한다. 단, 파일 내용 작성(Write/Edit)은 AI가 직접 한다
   - 폴더/파일 생성 (`mkdir`, `touch`)
   - Docker 명령어 (`docker-compose up`, `docker build`, `docker exec` 등)
   - 서버 실행 / 재시작 (`./gradlew bootRun`, `java -jar` 등)
   - 패키지 설치 (`npm install`, `pip install` 등)
   - 시스템 설정 변경 (`chmod`, `chown`, `systemctl` 등)
   - 그 외 모든 쉘 명령어
8. **설명은 항상 추가한다** — 이론 정리, 코드, 구조 변경 등 무엇을 하든 다른 AI나 사람이 봐도 왜 이렇게 했는지 이해할 수 있도록 충분한 설명을 함께 작성한다. 단순히 결과물만 만들지 않는다
9. **AI 전환을 고려한다** — 이 레포는 여러 AI가 이어받아 작업할 수 있다. HOW_WE_WORK.md, CONTEXT.md, context/contextNN.md를 항상 최신 상태로 유지해서 새 AI가 컨텍스트 없이도 동일한 방식으로 작업할 수 있게 한다

## 결정 로그 작성 규칙

이 레포에서 내린 주요 결정들은 반드시 이유와 함께 기록한다.
새 AI나 사람이 봤을 때 "왜 이렇게 되어 있지?" 라는 의문이 생기지 않아야 한다.

### 언제 기록하나

- 구조/폴더/파일 명명 방식을 결정했을 때
- 플로우나 규칙을 새로 만들거나 바꿨을 때
- 기술 선택 (라이브러리, 전략, 패턴)을 결정했을 때
- 사용자와 논의 끝에 방향을 결정했을 때

### 어디에 기록하나

- **context/contextNN.md "결정한 것들" 섹션** — 그날 논의에서 나온 결정을 여기에만 기록
- HOW_WE_WORK에는 결정 목록을 쌓지 않는다. 규칙(how)만 여기에 두고, 실제 결정 내용(what/why)은 context에 쌓는다

### 형식

```
- [날짜] 결정 내용 — 이유
```

예:

```
- [2026-04-07] Phase 번호를 0-A/0-B → 1~30 순번으로 변경 — 폴더명과 ROADMAP 번호가 불일치해서 새 AI가 매핑하기 어려웠음
- [2026-04-07] AI가 터미널 명령어 직접 실행 금지 — mkdir 등을 AI가 실행하면 사용자가 무슨 폴더가 생겼는지 모르고 넘어가는 문제 발생
```

## 새 AI가 이 레포를 처음 볼 때

> 이 레포를 처음 접하는 AI는 아래 순서로 읽어라

1. `CONTEXT.md` — 파일 읽기 순서 가이드
2. `context/contextNN-날짜.md` — 가장 최근 컨텍스트 (현재 상태, 완료된 것, 다음 할 것)
3. `HOW_WE_WORK.md` — 진행 방식 전체 규칙 (지금 이 파일)
4. `ROADMAP.md` — 전체 학습 로드맵 및 진행 현황
5. `notes/pending.md` — 아직 정리 못 한 개념 목록

**이 레포의 핵심 철학:**

- 단순 학습 예제가 아닌 실제 서비스(pong-to-rich)를 만들면서 모든 Phase를 체감한다
- 이론만 정리하거나 구현만 하지 않는다. 반드시 이론 → 구현 → 성능 비교 → devlog 기록 순서로 진행한다
- 처음부터 최적화하지 않는다. 느려지는 지점을 직접 체감하고 단계적으로 개선한다
- 모든 작업은 devlog에 기록한다. 막혔던 것, 에러 로그, 해결 과정 모두 남긴다
- AI는 백엔드 / 인프라 / AI 전 영역의 전문가 수준으로 대응한다
